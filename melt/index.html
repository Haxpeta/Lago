melt 工作流程

首先parser读取源文件,得到初始数据,将其存于自己的 resr 域中.
根据具体的设计可能对初始的原post进行修改与提炼.

然后交由post对数据进一步处理,得到完整的 post sxml 数据以及相关的原始文件信息
以及关于post的一些属性, 类似 tag, categories 这类.
得到post之后, 这个就算最初始的处理了. 对于后面的过程来讲, 最小的数据单元就是这个 post 了.

post 包含了由解析器处理后的sxml数据以及相关的文件,post属性

在得到一系列或者单个post之后,就会根据site的设定用每个独立的page
单元调用依据对应不同部分的renderer来对原始sxml进行渲染和组装处理.
这时候可能是输入诸如post时间, post分类, 生成相关的url这类数据的阶段.
组装完成后的数据会存放在page的cont域中.

最后最后, 我们可能得到了一系列或者单个page原数据.
这时最小的数据单元就是一个个或者单个page了.
依据site的设定一个个的输出到对应的目标位置. 在这期间是否会有什么其他的操作不得而
知, 但是基于sxml的特性, 依旧存在着继续处理的可能性.(因为sxml和一个lisp代码没有本质的区别,只是把它当成数据来处理了,这也是当初决定拿scheme来写这个东西的一个原因)

结尾时, 再把一些网站的资源文件搬到对应的地方去, 由asset来处理.

至此, 整个建站过程完成. 下面针对一些细节再次讨论一下:

  • 整个过程似乎没有提到一些特殊的页面, 比如categories和最开始的index页面, 这些怎么生成

    现在补充的一个信息就是, 以上的工作流程是命令 build 的全部工作流程.
    这里除了post可能会被写死专门处理有关post的事情, 其它的page,parser
    和site等等都是一些个带有特定意义的域的数据结构罢了. 具体的处理流程会由命令来
    决定. 针对这些特殊的页面可以专门为其写入一个原始sxml文件中, 然后依旧交给parser
    和page进行处理, 根据对应的配置输出到相应的位置.

同样的, 对于atom格式的RSS,可能的订阅服务等, 都可以交由那两个来处理.

所有的parser都是针对文件来说的, flax会针对不同格式的文件调用不同parser
从而得到相对应输出文件.
所有的post都是针对parser来讲的. 这里我其实也没有把post写死, 他也不一定只是针对
post来说的, 之后或许可以改个名.
然而page可能是对post也可能是对于parser来说的. page 是网站的最小
构成单元.

  • 我看着你说的东西不仅模糊, 而且我似乎也没有看到什么让我兴奋的点阿

    这样的, 由于flax还在构建中, 所以以上的说明也只是我目前的想法, 随时都有可能更改.
    我也要一个具体的思路才能继续构建. 具体的流程也会慢慢完善细致.

从我的角度来看, 我觉得兴奋的点还是存在的. flax 目前的目标是定义一套构建个人博客的
工具, 并且一直在努力减少原先代码的存在感. 意思就是让用户能够定义几乎所有的东西.
包括写博客时的工作目录的结构, flax的内置命令. 这些不需要修改flax的源码就能做到.
这个东西其实是很有限的, 但是如果把想象力交给用户的话, 一切都不一样了.

目前build命令其实做的事情就是执行用户的.scm文件(对, 就是执行), 然后用户在这个
文件里能够访问所有的实现特性以及引入库. 主题什么的就是一个程序. 这个程序可能会替代原先
默认的一些设置.

然后, 用户可以自定义melt的二级命令, 比如达到Hexo new "blog name"这种效果.

melt 改进计划以及详细介绍

目前 melt 的目标是能够最大限度的给予用户自由并且提供充足的 api 来减轻
用户的负担.

reader –> parser(可能对源文件做一些改变,将结果存储到 cache 中)

用于读取原 post 并把它转化成 sxml
现在 parser 将源文件编译,获取最原始数据,比如 metadata 和 content 这些.
最好是将这些东西具体存储在某些地方,目前来说最好的就是 chain 里的 data 域中.
对于一些不是 post 的东西也可以用它来 parse.

parser 可能的域:

  • type ==> 类别,用于识别源文件. (是一个点对 前面的是一个符号,后面的是一个识别程序)
  • proc ==> 用于处理源文件,提取原始信息,将其转化成 sxml.
  • resr ==> 数据,用来存放提取的信息. 不仅仅是 metadata 和文件内容,还包括更加基础的信息.
  • refp ==> 精炼函数,处理源文件,更 新它的信息.

其本身可能会有多段的分析组合

post –> 扩大

用于构建基本的 post 组件

现在将把 metadata 转换为具体的一些东西,用作内部构建使用

现在考虑一个处在parser和page之间的更加灵活的存在

可能的域:

  • meta ==> 文件的信息,不是 post 的信息. 比如修改时间,具体的路径文件名.
  • cont ==> 处理后的 post 内容
  • attr (属性) ==> 用来解释其在博客中的一些关系,包括该文的类型,分布.

此时 post 和 parser 组成了整个前端,完成一个源文件的分析更新工作.

renderer : 是 page 的一个组成部分

渲染, 针对特定的 page 进行特定的处理,对 post 的数据进行封装

renderer 可能的域:

  • type ==> 限定 rendre 对 page 的接口.
  • proc ==> 具体的处理函数,由 process 组成,可能嵌套 (可以用 chain 来实现)
  • data ==> 返回的数据, 用来与 page 对接,包含了一些 post 的数据

page

用于将每个 post 渲染后写入目标文件

现在将其拆分并且扩充它的职能.

可能的域:

  • cont ==> 完整的页面
  • comt ==> 页面的组成部分, 每个可能都会有一个 render 来渲染
  • proc ==> 具体的页面写入函数 (这个一般只有一个而且固定的)
    页面渲染首先是渲染部件,然后再组合,comt 由 render 来渲染,cont 由具体的 page 的 proc 来处理

site 对博客的整体配置

目前对他的作用感到迷茫, 感觉之后可以用作整个博客的架构,但是目前没有起到什么作用

可能的域:

  • layout ==> 网站布局, 是具体的文件集对网站组成部分的对应关系.
  • comt ==> 组件,比如个人介绍页, 资源页等,网站的组成. 构建方式是由一个个组件放入layout中
  • attr ==> 具体的文件配置属性

hook 系统 (有必要继续扩展的),甚至可以基于这个构建出那种响应式的系统

一定要物尽其用,改进后的这个系统相对于原来的基于 site obj 的系统优势很大
有必要继续扩大其作用. 仔细想想.

trigger 系统

触发系统, 更加强大

命令子系统

开发额外的插件, 提供额外的命令, 不一定只有 build 命令,目前已初步完成.
